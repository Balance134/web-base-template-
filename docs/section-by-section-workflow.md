# Section-by-Section Workflow

Главный принцип сборки сайта в этом проекте: **по одной секции за раз**.
Не "сделай весь сайт", не "сделай все 5 секций сразу".

## Почему именно так

Когда LLM генерит весь сайт за один проход — каждая секция выглядит
оторванно, потому что у модели нет якоря стиля. Получается коллаж
из разных эстетик.

Когда сначала делаем **одну хорошую секцию** (обычно Hero), а
следующие просят сделать **в том же стиле** — модель опирается на
существующий код как на эталон. Спейсинги, типографика, цвета,
характер анимаций сохраняются. Сайт становится цельным.

## Полный цикл

### Шаг 0: Настройка проекта
- Stack-инициализация (Next.js, Tailwind, shadcn/ui)
- Базовые токены в `globals.css` (можно стартовые, потом скорректируем
  под бренд клиента)
- Подключение Iconify в `layout.tsx`
- Подбор пары шрифтов через `next/font`

### Шаг 1: Hero-секция

Hero задаёт визуальный язык всего сайта. Делаем её **первой и
тщательно**.

**Что нужно для запуска:**
- Референс-скриншот (из Mobbin / Supahero / Stitch / макета клиента)
- Контекст бизнеса клиента (тематика, тон, ниша)
- Бренд-материалы клиента (логотип, цвета — если есть)

**Промпт-шаблон для Claude Code (Hero):**

```
Создай Hero-секцию для [тип бизнеса, например: "бутик-отеля 
в Самарканде"] на основе референса в [путь к скриншоту].

Контекст бизнеса:
- Название: [...]
- Тон: [премиальный сдержанный / тёплый дружелюбный / минималистичный]
- Целевая аудитория: [...]

Прежде чем кодить:
1. Прогони референс через чек-лист анализа из docs/design-system.md
2. Покажи мне извлечённую палитру и шрифты, я подтвержу
3. Покажи план секции (структура, что где располагается)

Когда подтвержу — генерируй код:
- Файл: components/sections/hero.tsx
- Стек: Next.js + Tailwind + shadcn (см. CLAUDE.md)
- CSS-переменные для цветов (через :root в globals.css)
- Iconify для иконок (Solar Linear)
- Контент на русском, релевантный бизнесу — без Lorem Ipsum
- Адаптивность mobile-first

Не делай пока другие секции. Только Hero.
```

**Что согласовываем после первой генерации:**
- Композиция (где заголовок, где CTA, где медиа)
- Цвета и шрифты (если не попал — корректируем токены)
- Анимация (есть/нет/насколько активная)

**Итерируем 2-4 раза, пока не закрепим стиль.**

### Шаг 2: Структурирование

После Hero — даём Claude команду закрепить общую архитектуру:

```
Hero готов и согласован. Теперь:

1. Убедись, что страница имеет чёткую семантическую структуру:
   <header> для навигации, <main> для контента, <footer> для подвала
2. Hero лежит внутри <main> как <section id="hero">
3. Создай пустой <Header> и <Footer> компоненты с базовой 
   структурой, чтобы они были готовы к наполнению
4. Добавь общий <Container> компонент для согласованной ширины 
   контента по всем секциям

Не добавляй другие секции. Только структура.
```

Это даёт правильную "коробку", в которую дальше будут добавляться секции.

### Шаг 3: Следующая секция

Выбираешь следующую важную секцию (для отеля — обычно "Типы номеров").
Снова находишь референс или используешь Stitch.

**Промпт-шаблон для последующих секций:**

```
Добавь секцию [название, например "Типы номеров"] на основе 
референса в [путь].

Принципы:
- Стилистически опирайся на существующий Hero. Используй те же 
  CSS-переменные, тот же характер spacing, те же шрифты
- Структуру и идею бери из референса, но адаптируй под наши 
  токены и контекст бизнеса
- Контент: 3-4 единицы (типа номера / преимущества / услуги — 
  что подходит секции)
- Файл: components/sections/[имя].tsx
- Импортируй секцию в app/page.tsx

Не трогай другие секции и существующие компоненты.
```

### Шаг 4-N: Остальные секции

Повторяем шаг 3 для каждой следующей секции.

**Стандартный набор секций для большинства лендингов/многостраничников:**
1. Hero
2. Преимущества / "О нас" коротко
3. Главное предложение (для отеля — типы номеров; для ресторана — меню;
   для услуг — каталог)
4. Социальное доказательство (отзывы / клиенты / награды)
5. Локация / контакты / карта
6. Финальный CTA + форма
7. Footer (полноценный)

### Шаг финальный: Полировка

Когда все секции на месте:
- Проверка адаптивности на 3 ширинах (375px / 768px / 1280px)
- Проверка accessibility (alt у картинок, aria-label у кнопок без текста)
- Lighthouse-аудит (Performance, SEO, Accessibility должны быть 90+)
- Подключение реальных изображений клиента (если до этого были
  плейсхолдеры)
- Заполнение метатегов (title, description, OG-image)

## Anti-patterns (так НЕ делаем)

### ❌ "Сделай сайт целиком за один промпт"

Получим набор разрозненных секций. Каждая будет выглядеть как сделанная
отдельным дизайнером.

### ❌ "Перепиши сразу 5 компонентов"

Когда уже что-то работает — изменения вносим точечно, по одному. Иначе
сложно отследить, что именно сломалось.

### ❌ "Скопируй референс пиксель в пиксель"

Цель — взять **структуру и приёмы**, а не точную копию. Точная копия
всегда смотрится хуже оригинала, потому что контекст и контент другие.

### ❌ "Доверяй промпту полностью, не проверяй"

После каждой секции открываем dev-сервер, смотрим вживую, итерируем.
LLM ошибается — и часто. Цена ошибки на одной секции маленькая, на
целом сайте — большая.

## Где применим Stitch

Stitch (от Google Labs) работает в этом workflow как **источник
референс-макетов** для конкретных секций.

Удобный сценарий:
1. Описываешь Stitch секцию ("hero для бутик-отеля в Самарканде с
   восточным влиянием, минимализм")
2. Stitch генерит макет
3. Корректируешь его прямо в Stitch (теперь это можно)
4. Через MCP-коннектор передаёшь макет в Claude Code
5. Claude Code превращает макет в Next.js-код по правилам этого
   проекта (CLAUDE.md + design-system.md)

Stitch заменяет Figma в твоём workflow — это сэкономленный шаг,
не теряя качества передачи дизайна в код.
