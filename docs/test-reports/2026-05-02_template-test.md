# Отчёт по тестовому прогону шаблона

**Дата:** 2026-05-02
**Проект:** `client-test` — фиктивный «Silk Route Boutique Hotel» в Самарканде
**Цель:** прогнать шаблон end-to-end до Hero включительно и зафиксировать
расхождения между документацией и реальностью.

---

## 1. Что сработало без проблем

- **Next.js App Router + TS strict + Tailwind v4.** Скаффолд через
  `create-next-app`, страница `/[locale]` поднялась с первого раза.
  TypeScript строгий, типы web-component'а Iconify через
  `types/iconify.d.ts` — рабочий паттерн, ошибок нет.
- **next-intl 4.x.** Роутинг (`i18n/routing.ts`), middleware/proxy и
  `getRequestConfig` склеились по доке next-intl, локали `ru`/`en`
  переключаются через `/ru` ↔ `/en`. `NextIntlClientProvider`
  обернул всё дерево, `useTranslations` работает и в server-, и в
  client-компонентах.
- **Design tokens в `globals.css`.** Палитра «бутик-отель премиум»
  (тёплый крем + глубокий изумруд + медь) — в `:root` как
  `R G B` тройки, экспонируются в Tailwind через `@theme inline` и
  утилиты `bg-background`, `text-foreground`, `border-border`,
  `bg-primary` сразу заработали.
- **Iconify через `next/script` (`afterInteractive`).** Один тег в
  `app/[locale]/layout.tsx` — иконки Solar Linear отрисовываются
  по всей странице. Ничего лишнего ставить не пришлось.
- **next/font (Fraunces + Inter).** Подключение через CSS-переменные
  и проброс в `<html className>` — без FOUT, без сюрпризов.
- **Hero как чистая типографика.** Получилась сильная композиция без
  фото вообще: H1 в Fraunces, eyebrow uppercase с трекингом, два
  радиальных свечения (медь сверху-справа, изумруд снизу-слева)
  как намёк на закат, тонкая горизонтальная линия — отсылка к
  горизонту/пустыне. Подтверждает: премиум-бутик может обходиться
  без иллюстраций до момента, когда придут реальные фото клиента.

## 2. Что нужно поправить в документации шаблона

### 2.1. Stock-фото запрещены полностью (КРИТИЧНО)

**Симптом:** при попытке поставить нейтральное фото с Unsplash для
Hero — Unsplash дважды подряд отдал не ту страну: сначала
Шефшауэн (Марокко), потом Берлин (Германия). По запросам
«Samarkand», «Registan», «boutique hotel» алгоритм Unsplash
возвращает «восточный/архитектурный вайб» без географической
точности.

**Риск:** в живом клиентском проекте это означает поставить чужую
страну на главную страницу отеля. Грубейшая ошибка, которая может
не пройти даже первый просмотр клиентом, но в худшем случае
утечёт в продакшен.

**Что исправлено в шаблоне:**
- `CLAUDE.md` (root + per-project) — пункт «Изображения»
  переписан: stock запрещён, разрешены только реальные фото
  клиента.
- `docs/section-by-section-workflow.md` (полировка) — убрано
  упоминание плейсхолдеров.
- `docs/hotel-domain-knowledge.md` (раздел «Если фото плохие») —
  явно разрешены фото с Booking (это реальные фото отеля),
  явно запрещён Unsplash/Pexels/Pixabay.
- `next.config.ts` — `remotePatterns` для `images.unsplash.com`
  удалён.

**Правило:** если реальных фото клиента ещё нет — собираем секцию
**без** фото (типографика, градиенты, абстракция). Это допустимая,
часто более сильная альтернатива «лишь бы какой-то картинки».

### 2.2. Tailwind v4 — документация написана под v3

**Симптом:** `docs/design-system.md` и CLAUDE.md описывают токены
через `@layer base { :root {...} }` и подразумевают `tailwind.config.js`.
`create-next-app` из коробки сейчас ставит **Tailwind v4**, где:
- нет `tailwind.config.js` (или он опционален);
- токены пробрасываются в утилиты через `@theme inline { --color-*: ... }`
  прямо в CSS;
- нет `@tailwind base/components/utilities` — только `@import "tailwindcss"`.

**Что сделано на проекте:** `globals.css` адаптирован под v4 —
сначала переменные в `:root` как RGB-тройки, затем `@theme inline`
переотображает их в `--color-background`, `--color-primary` и т.д.,
из которых Tailwind генерирует утилиты `bg-background`, `text-primary` и пр.

**Что нужно поправить в шаблоне:**
- В `docs/design-system.md` дописать раздел «Tailwind v4 vs v3»
  с готовым шаблоном `globals.css` под v4 (как в этом проекте).
- В CLAUDE.md уточнить: «Tailwind v4 (utility-first через
  `@theme inline`)».
- Решить вопрос пиннинга версии: либо в шаблоне зафиксировать v4
  и переписать доку, либо в `package.json` пиннить v3 и доку
  оставить. Рекомендация — **зафиксировать v4** (свежие проекты
  всё равно поедут на v4, доку переписать один раз).

### 2.3. Next.js 16: `middleware` → `proxy`

**Симптом:** при запуске dev-сервера Next 16 выдал предупреждение:

> The "middleware" file convention is deprecated. Please use "proxy" instead.

Файл `middleware.ts` в next-intl-сетапе работает, но deprecated.
В этом проекте оставлен как есть (тестовый прогон), но в шаблоне
надо обновить.

**Что нужно поправить в шаблоне:**
- Переименовать `middleware.ts` → `proxy.ts` (либо проверить,
  что next-intl уже обновил рекомендации).
- В CLAUDE.md / docs — упомянуть, что для Next 16+ файл называется
  `proxy.ts` (с тем же содержимым).
- Решить, фиксировать ли `next@^15` в шаблоне или сразу обновить
  до 16.

### 2.4. shadcn/ui — оверхед для Hero

**Симптом:** документация (`CLAUDE.md` → Stack) включает shadcn/ui
как часть стандартного стека и предлагает ставить его сразу. Для
этого тестового прогона до Hero включительно shadcn оказался
ненужным:
- из shadcn-компонентов в Hero пригодилась бы только Button —
  один тег `<a>` или `<button>` с Tailwind-классами решает то же
  и проще;
- `init` shadcn под Tailwind v4 + React 19 на момент 2026-05 ещё
  не везде гладкий, добавляет точку отказа на ровном месте;
- `clsx` + `tailwind-merge` (зависимости shadcn) для одного Hero
  не нужны — собственный `cn()` в `lib/utils.ts` (12 строк) решает
  задачу.

**Что нужно поправить в шаблоне:**
- В CLAUDE.md перенести shadcn из обязательной части стека в
  раздел «По требованию»: **ставим только когда нужны сложные
  компоненты** (Dialog, Command, Sheet, Form-обёртки, DataTable).
- В `docs/section-by-section-workflow.md` явно сказать: для
  Hero/Features/Pricing/Footer обходимся Tailwind + Iconify,
  shadcn подключаем когда впервые понадобится Dialog или Form.
- Включить минимальный `cn()` в `lib/utils.ts` шаблона по умолчанию
  (есть в этом проекте) — он закрывает 90% случаев без зависимостей.

## 3. Что улучшить в section-by-section workflow

### 3.1. Стартовый чек-лист «есть ли фото клиента»

Сейчас Stage «Working with new client project» в CLAUDE.md спрашивает
про источники материала, но не делает явный шаг **«получить и
сохранить фото в `/public/images/`»**. Из-за этого в моменте сборки
Hero возникает соблазн «пока поставлю что-нибудь со стока» — что и
случилось в этом прогоне.

**Предложение:** перед стартом Hero — обязательный sub-шаг:
1. Скачать топ-10 фото с Booking-страницы клиента (через парсер из
   `/scripts` или вручную).
2. Сложить в `/public/images/` с осмысленными именами
   (`courtyard-evening.jpg`, `room-deluxe-01.jpg`).
3. Если фото нет или они плохие — **сразу** решаем: ждём фото
   или собираем без фото. Со стоков не берём.

### 3.2. Hero без фото — допустимая ветка по умолчанию

Сейчас workflow подразумевает, что у Hero есть «герой-снимок».
Этот прогон показал, что **типографический Hero** — полноценная
ветка, которая может быть даже сильнее: премиум-ощущение от
типографики, отсутствие конкуренции с фотографиями нижних
секций, мгновенная адаптация под любого клиента до момента, когда
придут реальные фото.

**Предложение:** в `docs/section-by-section-workflow.md` в
секцию про Hero добавить две ветки:
- **A.** Hero с реальным фото клиента (классика).
- **B.** Hero типографический (когда фото ещё нет / низкого
  качества / клиент в процессе фотосессии). Даём пресет:
  кремовый фон, два радиальных свечения, тонкая линия-горизонт,
  крупный H1 в display-шрифте.

### 3.3. Контракт «один промпт — один шаг»

В этом прогоне всё держалось на жёстком разделении: Этап 1 →
Этап 2 → … → Этап 5. Это сработало. Стоит закрепить в шаблоне
правило (оно уже есть в `~/.claude/CLAUDE.md` для spec/plan/impl,
но не для секций):

**Один Hero за один заход.** Не «давай заодно подсуну Features».
Даже если время есть — следующая секция стилистически опирается
на _утверждённый_ Hero, не на черновой.

### 3.4. Скриншот-протокол

В этом прогоне dev-сервер поднялся, но сделать скриншоты на 375px
и 1280px из CLI-окружения (WSL без headless-браузера) штатно
нельзя. В шаблоне стоит:
- либо добавить `playwright` как dev-зависимость + `npm run shot`,
  который делает обе ширины;
- либо явно прописать в workflow, что скриншоты делает **разработчик
  вручную** через DevTools перед презентацией клиенту, и Claude в
  отчёте описывает композицию словами.

## Итог

Шаблон **рабочий**: за один сеанс прошли от пустой папки до
живого Hero на двух языках с design tokens, типографикой,
i18n-роутингом и Iconify. Серьёзное замечание одно — **Unsplash
запрещён** (исправлено в коде и доках). Остальные правки — уточнения
документации (Tailwind v4, Next 16, shadcn по требованию) и
расширение workflow (типографический Hero как самостоятельная
ветка, явный шаг про фото клиента).

Готов к следующему клиенту. Не пушим, не деплоим — это была
проверка шаблона.
